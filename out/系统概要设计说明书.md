# 谁是卧底系统概要设计说明书

## 1. 引言

### 1.1 编写目的

本文档用于说明“谁是卧底”项目前后端系统的总体设计方案，明确系统目标、技术架构、功能模块、数据结构、接口设计、核心业务流程及后续扩展方向。本文档可作为后续后端 SpringBoot 开发、前端 Vue 对接、课程答辩和项目维护的依据。

### 1.2 项目背景

“谁是卧底”是一款多人语言推理类游戏。传统玩法需要多名真人玩家同时参与，本项目在此基础上引入 AI 智能体能力，实现“1 名玩家 + 5 个智能体”的固定 6 人圆桌局。玩家可以与多个 AI 角色共同参与发言、投票、淘汰和结算流程，也可以将玩家席位托管给系统，观察智能体自动对局。

老师提供的后端学习资料中包含 Java 基础、SpringBoot 分层架构、内存存储、RestTemplate 调用 Dify Agent 以及 AI 词语查询模块设计。因此，本项目系统设计采用“Vue2 前端 + SpringBoot 后端 + Dify AI 服务”的整体方案。

### 1.3 设计依据

- 前端现有项目：Vue2、Vue Router、Axios、Element UI、LocalStorage 本地状态。
- 后端课程要求：Controller、Service、Entity、Store、Config 标准分层。
- AI 调用要求：SpringBoot 通过 RestTemplate 以 blocking 模式调用 Dify `/v1/chat-messages` 接口。
- 数据封装要求：统一使用 `ApiResponse<T>` 返回结构。
- 游戏业务要求：固定 6 人、人机对局、词语生成、发言、投票、淘汰、胜负判断、结算复盘。

## 2. 系统总体说明

### 2.1 系统名称

谁是卧底 AI 人机对局系统

### 2.2 系统目标

系统主要目标如下：

1. 支持用户登录进入游戏，不做复杂账号体系，优先满足课程项目演示。
2. 支持固定 6 人圆桌局，默认 1 名真人玩家和 5 个 AI 智能体。
3. 支持配置智能体名称、托管模式、卧底数量、白板数量、发言时间和智能体强度。
4. 支持通过后端调用 Dify Agent 自动生成平民词、卧底词和难度。
5. 支持游戏主流程：开局、查看词牌、发言、AI 发言、投票、淘汰、下一轮、结算。
6. 支持统一接口返回和前后端分离，便于后续扩展为真实联网对局或数据库持久化。

### 2.3 用户角色

| 用户角色 | 说明 |
| --- | --- |
| 真人玩家 | 进入系统并参与对局，可发言、投票、查看个人词牌 |
| 托管玩家 | 真人玩家将自己的席位交由系统托管，系统自动推进发言和投票 |
| AI 智能体 | 系统内置的虚拟玩家，具备不同风格特征，参与发言和投票 |
| 系统管理员 | 课程项目阶段暂不单独实现，可后续用于维护词库、模型配置和日志 |

## 3. 总体架构设计

### 3.1 架构模式

系统采用前后端分离架构：

```text
浏览器 Vue 前端
    |
    | HTTP/JSON
    v
SpringBoot 后端服务
    |
    | RestTemplate POST
    v
Dify Agent / 大模型服务
```

前端负责页面展示、用户交互和接口请求；后端负责游戏规则、房间状态、玩家状态、AI 词语查询和统一响应；Dify Agent 负责生成适合“谁是卧底”玩法的相近词语。

### 3.2 技术选型

| 层次 | 技术 | 作用 |
| --- | --- | --- |
| 前端框架 | Vue2 | 构建单页面应用 |
| 路由管理 | Vue Router | 管理登录、首页、设置、游戏、结算页面跳转 |
| 网络请求 | Axios | 发送 HTTP 请求到后端 |
| UI 辅助 | Element UI、原生 CSS | 表单、按钮、布局和交互样式 |
| 状态缓存 | LocalStorage | 保存登录态、设置项和临时游戏状态 |
| 后端框架 | SpringBoot | 提供 REST API 和业务服务 |
| JSON 处理 | Jackson ObjectMapper | 解析 Dify 返回 JSON 和业务 JSON |
| HTTP 客户端 | RestTemplate | 后端调用 Dify Agent 接口 |
| 简化实体 | Lombok | 自动生成 getter/setter、构造方法等 |
| 内存存储 | ConcurrentHashMap | 课程阶段替代数据库保存房间和玩家数据 |
| AI 服务 | Dify Agent | 生成平民词、卧底词和难度 |

### 3.3 系统分层

后端采用标准 SpringBoot 分层：

```text
com.undercover
├── controller   接口层，接收前端请求并返回 ApiResponse
├── service      业务层接口，定义游戏、房间、玩家、词语业务规范
│   └── impl     业务层实现类，处理核心逻辑
├── entity       实体层，定义 Player、GameRoom、WordVo 等数据模型
├── store        内存存储层，统一管理房间、玩家、发言、投票数据
├── config       配置层，处理跨域、RestTemplate、全局异常等配置
└── utils        工具层，统一响应、ID 生成、JSON 清洗等通用方法
```

前端采用页面、组件、接口、状态分离：

```text
src
├── views        页面视图：登录、首页、设置、智能体命名、游戏、结算
├── components   公共组件：弹窗、玩家席位、设置面板
├── router       前端路由和登录守卫
├── api          Axios 接口封装
├── store        本地状态和游戏状态模拟
└── assets       头像和静态资源
```

## 6. 核心业务流程设计

### 6.1 登录流程

```text
用户打开系统
  -> 输入用户名
  -> 前端保存登录态
  -> 跳转首页
```

课程阶段可使用简单登录，不需要数据库账号密码校验。后续如需扩展，可增加用户表和 token 鉴权。

### 6.2 创建对局流程

```text
前端提交游戏设置
  -> RoomController 接收请求
  -> RoomService 创建房间
  -> WordService 调用 Dify 生成词语
  -> GameService 创建 6 个玩家席位
  -> 分配平民、卧底、白板身份和词语
  -> GameMemoryStore 保存房间状态
  -> 返回 roomId 和初始化游戏状态
```

### 6.3 AI 词语生成流程

后端通过 RestTemplate 调用 Dify Agent，采用 blocking 阻塞模式一次性获取完整结果。

```text
WordController
  -> WordService.createWordByLevel
  -> 构造 Dify 请求体 inputs/query/response_mode/conversation_id/user
  -> 设置 Content-Type 和 Authorization
  -> RestTemplate.exchange 发送 POST
  -> 解析 Dify 外层 answer 字段
  -> extractJsonFromAnswer 清洗 AI 文本
  -> convertToWordVo 转换为 WordVo
  -> ApiResponse.success 返回前端
```

Dify 请求体核心结构：

```json
{
  "inputs": {
    "type": "词语生成"
  },
  "query": "难度：简单",
  "response_mode": "blocking",
  "conversation_id": "",
  "user": "abc-123"
}
```

请求头必须包含：

```text
Content-Type: application/json
Authorization: Bearer app-xxx
```

注意：`Bearer` 后必须保留空格，否则会出现 401 鉴权失败。

### 6.4 发言流程

```text
前端提交发言内容
  -> GameController.speak
  -> GameService 校验房间、玩家、阶段
  -> 保存 SpeechLog
  -> 切换下一个发言玩家
  -> 返回最新游戏状态
```

AI 智能体发言可以分两阶段实现：

1. 课程基础版：后端根据身份、词语、性格模板生成模拟发言。
2. AI 增强版：后端调用 Dify Agent，让模型根据当前上下文生成发言。

### 6.5 投票与淘汰流程

```text
前端选择投票目标
  -> GameController.vote
  -> GameService 记录投票
  -> 判断本轮是否投票完成
  -> 统计票数最高玩家
  -> 修改该玩家 alive = false
  -> 清空本轮 voted 状态
  -> 判断胜负
  -> 未结束则进入下一轮
```

投票统计规则：

- 每个存活玩家每轮只能投 1 票。
- 得票最高者出局。
- 若出现平票，可采用随机出局、无人出局或再次投票。课程阶段建议采用“平票无人出局，进入下一轮”，便于演示。

### 6.6 胜负判断流程

胜负判断建议规则：

| 条件 | 结果 |
| --- | --- |
| 所有卧底出局 | 平民阵营胜利 |
| 存活卧底数量大于等于存活平民数量 | 卧底阵营胜利 |
| 仅剩白板和卧底等特殊情况 | 按课程规则扩展 |

### 6.7 结算流程

```text
游戏结束
  -> GameService 写入 winner 和最终状态
  -> Result 页面请求结算数据
  -> 展示胜利阵营、MVP、平民词、卧底词、玩家身份、存活状态
```

## 8. 内存存储设计

课程阶段可暂不使用数据库，采用 `GameMemoryStore` 全局单例存储运行时数据。

建议结构：

```java
@Component
public class GameMemoryStore {
    private final ConcurrentHashMap<String, GameRoom> roomMap = new ConcurrentHashMap<>();
    private final ConcurrentHashMap<String, List<SpeechLog>> speechMap = new ConcurrentHashMap<>();
    private final ConcurrentHashMap<String, List<VoteRecord>> voteMap = new ConcurrentHashMap<>();
}
```

设计原则：

- 所有房间数据统一由 Store 管理，Service 层不直接创建全局集合。
- 使用 `ConcurrentHashMap` 保证多人并发访问时基本安全。
- 课程项目重启后数据丢失是可接受的；如需长期保存，后续可替换为 MySQL 或 Redis。

## 9. 安全与异常设计

### 9.1 跨域设计

后端可在 Controller 上使用 `@CrossOrigin`，也可在 `config` 包中统一配置跨域。建议后期统一使用 `CorsConfig`，避免每个 Controller 重复写注解。

### 9.2 参数校验

后端应校验以下内容：

- 房间号不能为空，且必须存在。
- 玩家必须属于当前房间。
- 发言内容不能为空。
- 投票目标必须是存活玩家。
- 卧底数量、白板数量不能超过玩家总数。
- 游戏结束后不允许继续投票或发言。

### 9.3 AI 调用异常

Dify 调用可能出现以下问题：

| 异常 | 处理方式 |
| --- | --- |
| 401 鉴权失败 | 检查 `Authorization` 和 `Bearer` 空格 |
| 404 地址错误 | 检查 Dify 接口地址 |
| 500 服务异常 | 返回友好错误信息，记录日志 |
| answer 缺失 | 返回“AI 返回格式异常” |
| JSON 解析失败 | 使用 `extractJsonFromAnswer` 清洗后再解析 |

## 10. 前后端对接设计

### 10.1 前端接口封装调整

当前前端已有 Axios 封装思想，后续建议统一为：

```text
src/api/request.js      axios 实例
src/api/user.js         登录接口
src/api/room.js         房间接口
src/api/game.js         游戏接口
src/api/word.js         词语接口
```

前端页面不直接写 URL，而是调用 api 文件中的函数，便于后续修改后端地址。

### 10.2 代理配置

开发阶段可在 `vue.config.js` 中配置代理：

```js
devServer: {
  proxy: {
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true,
      pathRewrite: { '^/api': '' }
    }
  }
}
```

前端请求 `/api/game/state`，由代理转发到 SpringBoot 的 `/game/state`。

### 10.3 本地模拟到后端接口迁移

当前前端 `store/game.js` 中存在本地模拟状态。迁移方案如下：

| 当前本地函数 | 后端对接接口 |
| --- | --- |
| `getGameState()` | `GET /game/state` |
| `saveGameState()` | 后端接口内部保存 |
| `resetGameState()` | `POST /room/create` 或 `POST /game/start` |
| 本地智能体发言数组 | `POST /game/agentSpeak` |
| 本地投票累加 | `POST /game/vote` |

## 11. 非功能性设计

### 11.1 可维护性

- 后端 Controller 只负责接收请求和返回结果，不写复杂业务逻辑。
- Service 层使用接口和实现类，方便后续替换规则。
- Entity 统一封装数据字段，使用 Lombok 减少模板代码。
- Store 统一管理内存数据，避免数据分散。

### 11.2 可扩展性

系统后续可扩展：

- 增加真实多人联机房间。
- 接入 WebSocket 实现实时发言和投票同步。
- 增加数据库保存历史战绩。
- 增加更多 AI 智能体性格和难度。
- 将 AI 发言、AI 投票也接入 Dify 或其他大模型。

### 11.3 易用性

- 前端保持固定 6 人圆桌，降低玩家理解成本。
- 词牌只显示给当前玩家，避免身份泄露。
- 首页提供继续上一局、规则说明、智能体说明等常用入口。
- 结算页直接公布身份和词语，方便复盘。

## 13. 总结

本系统以 Vue2 前端为交互入口，以 SpringBoot 后端为业务核心，以 Dify Agent 为 AI 词语生成能力来源。整体设计符合课程要求的分层架构：前端负责页面和交互，Controller 负责接口入口，Service 负责游戏规则，Store 负责统一内存数据，WordService 负责调用 AI 服务。

该设计既能支持当前“1 名玩家 + 5 个智能体”的课程演示版本，也保留了后续扩展真实多人联机、数据库持久化、AI 发言和 AI 投票的空间。
